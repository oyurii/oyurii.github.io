Tools Workflow

This document describes how to use scripts in Tools/ for access-code generation, encryption, and public-site build.

Directory Overview

  • Tools/group_list.md - source list of users (one per line).
  • Tools/generate_group_list.py - generates per-discipline code maps.
  • Tools/encrypt.py - prepares encrypted lesson content.
  • teaching/*.qmd - source lesson files (plaintext source lives only here).
  • .build_src/ - generated sanitized build workspace (temporary/build artifact).
  • .github/workflows/publish_site.yml - CI pipeline that builds from .build_src and publishes to oyurii.github.io.

Required Preconditions

  1. Lessons that need protected content must have front matter with:
    • encrypt: true (or encript: true for backward compatibility)
  2. Lesson must contain marked block:
    • <!-- ENCRYPTED_SECTION_START -->
    • <!-- ENCRYPTED_SECTION_END -->
  3. Tools/*_group_list.json files must exist for each encrypted discipline.

Script 1: generate_group_list.py

Generates per-discipline files with user codes.

Default run

cd Tools
./generate_group_list.py

What it does

  • Reads users from group_list.md
  • Detects encrypted disciplines from ../teaching/*.qmd
  • Creates/updates *_group_list.json
  • Keeps existing codes by default
  • Rotates only selected users if --people is provided

Parameters

  • --group-file GROUP_FILE
    User list file (default: group_list.md)
  • --courses-dir COURSES_DIR
    Directory with lesson .qmd files (default: ../teaching)
  • --output-dir OUTPUT_DIR
    Where to write group-list files (default: current dir)
  • --disciplines [DISCIPLINES ...]
    Explicit discipline names (skip auto-discovery)
  • --people [PEOPLE ...]
    Rotate codes only for selected users
  • --format {json,yaml}
    Output format (default: json)
  • --code-length CODE_LENGTH
    Random part length (without code- prefix), default 18

Examples

# Rotate codes only for yurii and roman
./generate_group_list.py --people yurii roman

# Generate only for specific disciplines
./generate_group_list.py --disciplines quantum-mechanics statistical-physics

Script 2: encrypt.py

Builds encrypted per-user lesson variants and sanitized public source.

Modes

A) Build mode (default, recommended)

cd Tools
./encrypt.py
# same as: ./encrypt.py --mode build

This mode:

  1. Recreates .build_src/ from project root.
  2. Generates per-user encrypted qmd files in .build_src/teaching/*_encripted.qmd.
  3. Sanitizes public .build_src/teaching/*.qmd so plaintext protected block is removed.
  4. Adjusts .build_src/_quarto.yml to allow rendering encrypted variants in build workspace.
  5. Marks per-user pages with search: false to keep them out of public search index.

B) Personal mode (manual/debug)

cd Tools
./encrypt.py --mode personal

This mode writes *_encripted.qmd into target content directory (default ../teaching). Use only when explicitly needed.

Parameters

  • --mode {personal,build}
    Execution mode (default: build)
  • --content-dir CONTENT_DIR
    Lesson source directory for personal mode (default: ../teaching)
  • --group-lists-dir GROUP_LISTS_DIR
    Directory with *_group_list.json (default: current dir)
  • --people [PEOPLE ...]
    In personal mode, generate only selected users
  • --suffix SUFFIX
    Suffix for encrypted variants (default: encripted)
  • --project-root PROJECT_ROOT
    Project root for build mode (default: ..)
  • --build-dir BUILD_DIR
    Build workspace path (default: ../.build_src)
  • --content-subdir CONTENT_SUBDIR
    Content subdir in build workspace (default: teaching)
  • --group-lists-subdir GROUP_LISTS_SUBDIR
    Group-list subdir in build workspace (default: Tools)

Full Recommended Workflow

From oyurii_project_private/:

  1. Update lesson content in teaching/*.qmd (including protected section markers).
  2. Update user list if needed in Tools/group_list.md.
  3. Regenerate/rotate codes:
cd Tools
./generate_group_list.py
  1. Build sanitized workspace:
./encrypt.py --mode build
  1. Render site from sanitized workspace:
cd ../.build_src
quarto render
  1. Publish only:
  • .build_src/_docs/

GitHub Actions Publishing

Current workflow file: .github/workflows/publish_site.yml

Pipeline steps:

  1. Checkout private repo.
  2. python3 Tools/encrypt.py --mode build
  3. quarto render in .build_src/
  4. Copy .build_src/_docs/* into oyurii.github.io/docs/
  5. Commit + push to public repo.

Important Security Notes

  • Do not publish source teaching/*.qmd from main tree.
  • Publish only .build_src/_docs.
  • In build mode, public pages do not contain plaintext protected content.
  • Per-user encrypted pages are generated separately and fetched only after user authorization.
  • Authorization UI is modal and opens only after user clicks Full access.

Search Behavior

  • Per-user encrypted pages are marked with search: false.
  • They are excluded from search.json indexing.

Troubleshooting

1) Invalid username or access code (Open the site via HTTP(S), not file://)

You opened pages directly from filesystem. Use HTTP server:

cd .build_src/_docs
python3 -m http.server 8000

2) Invalid username or access code (HTTP 404 ...)

Usually means encrypted page was not published or wrong path. Ensure you deployed full .build_src/_docs/teaching/ including *_encripted.html.

3) Quarto error: attempt to write a readonly database

Quarto cache issue in restricted environment. Re-run quarto render with proper permissions/environment (normal local shell usually works).

4) Search stopped working

Rebuild from .build_src and ensure final published search.json exists:

ls -la .build_src/_docs/search.json

Also verify search.json does not include encrypted per-user pages:

rg "_encripted\\.html" .build_src/_docs/search.json
# should return no matches

Quick Command Cheat Sheet

# 1) Generate/update user codes
cd Tools
./generate_group_list.py

# 2) Build sanitized workspace (default mode)
./encrypt.py

# 3) Render website
cd ../.build_src
quarto render

# 4) Local preview via HTTP (required for authorization fetch)
cd _docs
python3 -m http.server 8000
